We call this architectural style as Monolithic Architecture.

So, what typical challenges* are we facing?

*keyword is typical

Scalability

Monolitic application can scale; however, it is not really economical, i.e. scaling a particular service requires scaling other components as well.

Resiliency

The malfunctioning of certain component can easily tanks entire application. E.g. Thread pools and connection pools are often managed by application server and are shared by all components in entire application.

Deployment

When time goes by, it is unavoidable that a deployment requires to bring down components more than it is really required. Well, this will in-turn affect the time-to-market as we need to notify, get consensus, and testing with all parties.

Also, advance deployment methods such as A-B test deployment, green-blue deployment, and canary deployment are very difficult to be achieved with Monolithic Architecture.

Of course, if design (both organizational design and technical design) and DevOps are done right consistently and continuously, monolithic can be very scalable, resilient, easy to deploy with good time-to-market.

Look at StackExchange.com (master site of the bespoke StackOverflow.com) as an example!

p.s. In other words, if things are not done right, you will suffer 1000 times more from microservices than monolithic.

By definition...

Microservices are a software architecture that involves developing software applications or platforms as many finely grained services that can be independently designed, developed, tested and deployed.

*In my humble opinion, the keywords are designed and deployed.

How is this different than service-oriented architecture ("SOA")?

Although in both cases we are indeed talking about a set of services, the ambition* of these services is different.

Speaking about ambition...

SOA attempts to put these services forward to anybody who wants to use them.
Microservices, alternatively, are created with a much more focused and limited goal in mind, which is acting as a part of single distributed system.

Unlike with SOA, microservices often exist implicitly. Microservices are well-known to the consumers, and therefore do not require service description.

To summarize the SOA vs Microservices...

SOA was designed with an ambition to solve very complex enterprise architecture problems, with the goal of facilitating a high level of reusability.

In contrast, microservices architecture was embraced by companies attempting to scale a single web property to web scale levels, enable continuous evolution, make engineering teams more efficient, and avoid technology lock-in.

Organizational Chanlleges

Architectural evolution

Evolving to microservices from a traditional monolith is not always straightforward*, and probably require rearchitecture and replatforming to get there.

*It is always not straightforward, i.e. it is never straightforward.

Improving DevOps maturity

To be successful with microservices, teams need to improve the level of collaboration between development, test and IT operations.

Increased automation

To successfully deliver microservices, we need much higher degree of build, release, deployment and environment automation into the technology stack.

Upskilling your people (hard-skills)

Need investment, training and practice to deploy and operate a microservice-based system successfully.

MOST important of all is to cultivate your team to become full-stack developer

Need mental replatform and transformation... Microservices enables complete ownership by a self-contained and cross-functional single team, from definition through development, deployment, operations, and sunsetting
Technical Chanlleges

Expecting eventual consistency

This one is critical! ACID* is likely GONE! Project team MUST convince business owner to understand and accept the behavior of eventual consistency!

Some patterns will help, e.g. Compensating Transaction pattern, Event Sourcing pattern, etc.

*Data management of microservices is a HUGE and extrememly difficult topic that will be talked in separate sharing!

Design for failure

Entire microservice system is about moving parts, i.e. many services with many remote calls

Design patterns like Circuit Breaker, Bulk-head are there to help.

Test for failure

Do not forget to test your "design for failure"... Put resiliency testing into your testing pipeline.

Tools like IBM Gremlin / NetFlix Chaos Monkey can help.

You can't manage what you can't measure

Each microservice should have built-in performance data and metrics exposed.

Libraries like Spring Actuator, Dropwizard Metrics help.

Do not underestimate network latency

Cache aggressively (and manage them well).

Put performance testing into your testing pipeline.

Service Management

How can a service be discovered by others?

Do we need a dedicated service registry* (like Consul)?

Is the hosting environment (PaaS / CaaS) having built-in feature?

Server or client-side load-balancing?

Self-register (and deregister) on instance create and destroy?

*See Service Discovery section of here

Operational complexity

Again, all are moving parts and be prepared to have hundreds of instances. Each microservice should have built-in health-check (and exposed) and self-healing.

Also, we need different way to manage applicaiton log than traditional monolithic application, i.e. logs are usually treated as events and sent to a store for further analysis and monitoring.

Application Packaging

We used to package application as war / ear files and deploy them to application server like WAS.

In microservices, there is a new way to package your application!

Über Jar

It packages all dependencies and application code into a self-contained JAR with a flat class loader. This makes it easier to understand application startup, dependency ordering, and log statements; but more importantly, it helps reduce the number of moving pieces required to take an app safely to production.

All you need to run your application is

java -jar <name.jar>

That's it!

Instead of running an application in an application container, we now run an application container inside your application!

Woo!

Production Ready

All these frameworks come with built-in health data, performance metrics accessible via different ways.
